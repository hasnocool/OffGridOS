# 8. Credits, Incentives, and Token Economics

## Design objective

Incentives should increase useful participation: hosting a service, maintaining a location, sharing equipment, contributing verified knowledge, or helping with installation and support.

## Start with internal credits

The first implementation should use an auditable, centrally administered or community-administered credit ledger. Credits can be issued for verified actions and redeemed for defined goods or services. This avoids premature blockchain complexity and reduces transaction costs.

## Optional settlement layer

If a later community requires transferable settlement, the system can integrate an established low-fee network through a custodial or non-custodial adapter. The application must not require a particular chain, and token balances must never replace clear pricing, refunds, accounting, or consumer protections.

## Controls

Issuance rules, caps, expiry, fraud review, dispute handling, tax/compliance review, privacy, and conversion policies must be documented before launch. Credits should reward measurable service quality, not speculative trading or recruitment.

## Examples

Landowners may receive credits for verified overnight access or maintained amenities. Businesses may earn credits for charging, water, repair, food, or connectivity services. Contributors may earn credits for accepted documentation, moderation, or installation work.
